
## 1. 文档目的

本文档用于指导 Codex 进行第一阶段产品开发，目标是做出一个可运行的 AI 相机 MVP：

- 能理解当前画面中的主体、场景和拍摄意图
    
- 能在拍摄前/拍摄中给出实时建议
    
- 能在拍摄后自动选片、调色和生成成片
    
- 能形成从“识别-建议-拍摄-出片”的闭环体验
    

本文档同时覆盖：

- 产品定位
    
- 用户流程
    
- 核心功能设计
    
- 系统架构
    
- 关键技术实现
    
- 数据与模型方案
    
- 开发拆解
    
- MVP 范围与里程碑

## 2. 产品定位

### 2.1 产品一句话

一个面向普通用户的 AI 拍摄助手：不仅自动调色，还能实时理解画面，给出构图、机位、姿势和光线建议，提高出片率。

### 2.2 与现有产品的差异

现有手机相机和修图产品大多只覆盖某一段：

- 原生相机：强在自动成像优化
    
- 美颜/修图产品：强在拍后处理
    
- 少量 AI 相机功能：开始给局部提示，但还没有形成完整闭环
    

本产品的核心差异是：

1. **拍前理解**：理解是在拍人像、探店、旅行、夜景、自拍还是合影
    
2. **拍中指导**：实时给用户可执行的建议，而不是只做底层算法增强
    
3. **拍后成片**：自动选片、统一风格、直接输出可分享结果
    
4. **以出片率为目标**：不是给专业参数，而是帮助普通用户“拍得更好看”
    

### 2.3 初期产品边界

第一阶段不做“专业摄影控制台”，不做复杂手动参数系统，优先做：

- 人像 / 自拍 / 他拍
    
- 探店 / 街拍 / 旅行 / 夜景
    
- 单人和双人场景
    
- 静态画面为主，轻量支持动态抓拍

## 3. 目标用户与核心场景

### 3.1 目标用户

#### 用户 A：普通高频拍照用户

特征：

- 喜欢发朋友圈 / 小红书 / Instagram
    
- 没有系统摄影知识
    
- 想提升照片质感和稳定性
    

痛点：

- 不知道怎么站、怎么摆、怎么构图
    
- 同一场景拍 20 张，最后没几张能用
    
- 拍的时候看着还行，拍完发现不好看
    

#### 用户 B：情侣 / 朋友互拍用户

特征：

- 常见于出游、探店、打卡
    
- 一人给另一人拍
    

痛点：

- 不知道机位、角度、取景
    
- 背景杂乱、人像比例差
    
- 不知道现在该让对方怎么动作
    

#### 用户 C：轻内容创作者

特征：

- 对“风格统一”和“稳定出片”要求更高
    
- 追求一定审美感，但不一定会后期
    

痛点：

- 每次拍摄都要靠经验重复试错
    
- 拍后修图耗时
    
- 同一组图风格不统一
    

### 3.2 核心使用场景

1. **自拍**：AI 提示脸部角度、手机高度、背景整理、表情建议
    
2. **朋友互拍**：AI 提示摄影者站位、焦距建议、人物位置、动作建议
    
3. **探店 / 咖啡馆**：自动判断桌面、逆光、暖光环境，给风格建议
    
4. **旅行街拍**：自动判断建筑、街景、人景比例，给取景建议
    
5. **夜景**：提示稳定拍摄、人物站位、补光建议和曝光策略
    
6. **合影**：提示人物分布、遮挡、表情时机、自动多帧优选

## 4. 产品目标与指标

### 4.1 北极星指标

**有效出片率** = 用户认为“可直接分享”的照片数量 / 本次拍摄照片数量

### 4.2 一级指标

- 引导建议触发率
    
- 建议采纳率
    
- 拍摄后成片保存率
    
- 自动推荐照片采用率
    
- 用户单次会话平均出片数
    
- 次日 / 7 日留存
    

### 4.3 技术指标

- 实时建议平均延迟 < 300ms
    
- 拍摄预览帧分析稳定在 10~15 FPS
    
- 姿态估计成功率 > 95%（人像近景条件下）
    
- 场景分类 Top-1 准确率 > 85%
    
- 成片推荐点击率 > 40%

## 5. 产品功能设计

## 5.1 产品闭环

### 阶段一：拍前理解

输入：实时预览帧 + 设备信息 + 用户选择模式

输出：

- 当前场景标签（咖啡馆 / 夜景 / 室内 / 街拍 / 逆光 / 自拍）
    
- 主体识别（单人 / 双人 / 合影 / 食物 / 建筑）
    
- 当前问题识别（脸过暗 / 背景乱 / 构图偏 / 头顶裁切 / 角度不佳）
    
- 拍摄策略建议（建议广角 / 2x / 下蹲 / 靠窗 / 横构图等）
    

### 阶段二：拍中指导

在预览画面上叠加引导层：

- 人物框
    
- 黄金分割 / 中心构图辅助线
    
- 动作建议文字
    
- 机位箭头提示
    
- 光线建议提示
    

例如：

- “镜头再低一点”
    
- “人物往右移半步”
    
- “抬下巴，肩膀侧一点”
    
- “靠近窗边，脸会更亮”
    
- “背景过乱，建议缩小取景范围”
    

### 阶段三：拍后成片

拍照完成后自动：

- 多帧优选：闭眼检测、模糊检测、表情优选
    
- 自动修图：曝光、白平衡、肤色、局部增强
    
- 风格统一：根据场景使用风格模板
    
- 给出 3 张推荐图：原图增强版 / 风格版 / 社交分享版

## 9. 关键技术实现方案

## 9.1 场景识别

### 目标

识别当前拍摄场景，用于驱动后续建议和风格策略。

### 输入

- 预览帧缩略图（例如 224x224）
    
- 时间窗口内多帧统计特征
    
- 设备状态（前/后摄，焦段，亮度估计）
    

### 输出

- scene_type：indoor / cafe / street / travel / night / backlight / food / portrait / group
    
- confidence
    

### 实现方式

第一阶段采用轻量 CNN / ViT 分类器：

- MobileNetV3 / EfficientNet-Lite / MobileViT
    
- 转 ONNX / CoreML / TFLite 做端侧推理
    

### 技术细节

- 多标签分类优于单标签分类，例如夜景 + 人像 + 室内可同时成立
    
- 对连续帧做滑动窗口平滑，避免标签抖动
    
- 使用 hysteresis（滞后）机制减少 UI 闪烁
    

### Codex 开发要求

- 抽象接口：`predict_scene(frame) -> ScenePrediction`
    
- 支持 mock 模型和真模型切换
    
- 支持阈值配置化
    

---

## 9.2 主体识别

### 目标

识别当前是否以人、多人、食物、建筑为主体，并估计主体位置与占比。

### 实现方式

- 通用目标检测模型：YOLO-NAS / YOLOv8n / RT-DETR small
    
- 对人像优先优化：person / face / upper body / food / drink / landmark 等类别
    

### 输出

- 主体类别
    
- bounding boxes
    
- 人脸框
    
- 主体面积占比
    
- 是否存在遮挡
    

### 作用

- 判断应该做人像指导还是场景构图指导
    
- 判断人物是否过小/过大
    
- 判断是否需要建议变焦
    

---

## 9.3 姿态估计与动作建议

### 目标

对人物关键点进行检测，并映射到可执行的姿势建议。

### 实现方式

#### 基础层

- 使用轻量人体关键点模型，如 MoveNet / BlazePose / RTMPose-tiny
    
- 提取头部、肩部、手肘、手腕、髋部等关键点
    

#### 策略层

根据关键点规则生成提示：

- 肩膀完全平行 + 正面站立 -> 建议“身体侧一点”
    
- 手臂紧贴躯干 -> 建议“手离开身体一点”
    
- 下巴过低 -> 建议“抬一点下巴”
    
- 头顶空间不足 -> 建议“镜头上移或后退”
    
- 双腿并拢僵硬 -> 建议“前后错步站”
    

### 输出

- 姿态标签
    
- 建议动作文案
    
- 可视化引导锚点
    

### 关键点

第一阶段不要直接上大模型生成动作建议，而是：

- 用规则系统保障稳定性
    
- 用少量模板文本保证清晰可执行
    

### Codex 开发要求

- 规则引擎可配置：YAML / JSON
    
- 每条规则可独立开启/关闭
    
- 支持记录命中规则用于调试
    

---

## 9.4 构图评估

### 目标

评估当前画面构图质量，并输出可调整的建议。

### 特征来源

- 主体位置
    
- 人脸位置
    
- 地平线估计
    
- 边缘裁切情况
    
- 背景复杂度
    
- 前景/背景分离程度
    
- 主体占画面比例
    

### 方法一：规则式 MVP

规则示例：

- 人脸贴近边缘 -> “人物再往中间一些”
    
- 主体比例过小 -> “再靠近一些”
    
- 主体头顶被裁切 -> “镜头抬高一点”
    
- 背景杂乱度高 -> “缩小取景范围”
    
- 多人分布不均 -> “两个人站近一点”
    

### 方法二：学习式增强（后续）

- 构建构图评分模型
    
- 训练 aesthetic score / composition score
    
- 结合用户保留照片做偏好学习
    

---

## 9.5 光线评估与建议

### 目标

快速判断是否逆光、脸部过暗、光线方向不佳，并给出建议。

### 基础实现

- 统计整图亮度分布
    
- 估计人脸区域平均亮度
    
- 比较主体与背景亮度差
    
- 识别高光溢出和阴影过重区域
    

### 输出建议

- “脸有点暗，靠近亮处”
    
- “背景太亮，人物往左移动”
    
- “夜景环境建议保持稳定 1 秒”
    
- “建议打开补光/屏幕补光”
    

### 可增强方向

- 利用语义分割识别窗户/天空/强光源
    
- 估计主光方向，优化提示准确性

## 9.6 实时建议策略引擎

### 问题

同一时刻可能有很多问题：构图、光线、姿态、背景、表情。不能全部同时告诉用户。

### 方案

设计 Suggestion Policy Engine，对建议进行排序与去重。

### 优先级建议

1. 拍不了 / 看不清类：严重逆光、人物太小、模糊风险
    
2. 出片率强相关类：构图问题、脸部角度、头顶裁切
    
3. 审美增强类：动作优化、风格建议
    
4. 次要提示：细节修饰
    

### 策略原则

- 同屏最多展示 1 条主建议 + 1 条次建议
    
- 2 秒内不频繁切换
    
- 连续 3 帧以上稳定才提示
    
- 用户已采纳后暂时抑制重复提示
    

### 示例接口

```python
class SuggestionEngine:
    def rank(self, scene, subject, pose, composition, lighting, history):
        ...
        return primary_suggestion, secondary_suggestion
```

## 9.7 最佳照片选择（Best Shot）

### 目标

拍摄结束后从多帧中自动选出最优照片。

### 评分维度

- 清晰度
    
- 闭眼检测
    
- 表情自然度
    
- 人脸朝向
    
- 构图分数
    
- 曝光分数
    
- 抖动程度
    

### 实现方式

#### MVP

- 基于规则打分：每张图算一个总分
    
- 选择 Top-K
    

#### 后续增强

- 训练 ranking model
    
- 融合用户历史选择作为监督信号
    

### 伪代码

```python
def score_image(features):
    return (
        0.25 * features.sharpness
        + 0.20 * features.expression_score
        + 0.20 * features.eye_open_score
        + 0.15 * features.composition_score
        + 0.10 * features.exposure_score
        + 0.10 * features.face_angle_score
    )
```

## 9.8 自动增强与风格引擎

### 目标

基于场景与主体，输出“更可分享”的成片。

### MVP 方案

#### 基础增强

- 自动曝光校正
    
- 白平衡调整
    
- 对比度 / 饱和度 / 阴影提升
    
- 局部人脸提亮
    
- 基础磨皮与肤色保护
    

#### 风格模板

按场景提供风格预设：

- Cafe Warm
    
- Street Cool
    
- Night Neon
    
- Travel Fresh
    
- Portrait Soft
    

### 实现方式

- 第一阶段优先采用经典图像处理 + LUT 模板
    
- 复杂生成式重绘不纳入 MVP
    
- 通过参数模板实现低成本快速迭代
    

### 接口设计

```python
render_style(image, scene_type, style_id, face_regions) -> image
```

## 14. 非功能性要求

### 性能

- 预览分析不阻塞拍摄
    
- 弱机型自动降级分析频率
    
- 拍照主链路优先级最高
    

### 稳定性

- 任一模型失败时不影响基础相机使用
    
- 所有算法模块支持 fallback
    

### 隐私

- 默认端侧处理实时预览
    
- 未经授权不上传原图
    
- 埋点仅记录匿名特征和交互事件
    

### 可配置性

- 文案、阈值、规则、风格模板可远程配置
    

---

## 15. 风险与应对

### 风险 1：实时性能不够

应对：

- 预览帧降采样
    
- 降低分析频率到 5~10 FPS
    
- 多模型异步并行
    
- 仅在关键帧上做重分析
    

### 风险 2：建议过多引起干扰

应对：

- 强约束 suggestion ranking
    
- 仅展示最关键建议
    
- 加入提示节流机制
    

### 风险 3：建议不准确导致反感

应对：

- MVP 先做确定性强的规则
    
- 文案使用保守表达
    
- 增加“关闭建议”入口
    

### 风险 4：风格效果不稳定

应对：

- 先采用保守风格模板
    
- 保留原图增强版作为默认推荐

## 16. MVP 范围定义

### MVP 必须回答的问题

1. 用户是否愿意在拍摄时接受 AI 提示？
    
2. 实时建议是否真的提升出片率？
    
3. 用户更在意姿势指导、构图指导，还是拍后自动成片？
    
4. 什么场景下产品价值最明显？
    

### MVP 成功标准

- 有 30% 以上用户采纳过建议
    
- 自动推荐图被保存的比例明显高于原图直接保存
    
- 用户主观反馈认为“更会拍了”或“更容易出片”

## 19. 附录：建议文案模板示例

## 20. TL;DR

这不是一个“再做一个带滤镜的相机”，而是一个以“出片率”为目标的 AI 拍摄助手。

MVP 的关键不在于一次性做最强模型，而在于先跑通如下闭环：

**理解画面 -> 给出少量高价值建议 -> 拍摄 -> 自动挑出最好的一张 -> 输出可分享成片**

只要这个闭环成立，后续无论接更强模型、做个性化还是接生成式能力，都有明确演进路径。
